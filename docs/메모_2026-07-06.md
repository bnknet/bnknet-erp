- 원인: 셀을 숫자로 읽음. 수정: `XLSX.read(buf,{type:'array',cellDates:true})` + `fmtItemDate(v)`가 Date/serial/텍스트 모두 MM/DD로.
- **기존 잘못 저장분 정정 SQL(실행 완료 7/6)**:
```
update public.approval_items set item_date = to_char(date '1899-12-30' + item_date::int, 'MM/DD') where item_date ~ '^[0-9]{4,6}$' and item_date::int between 40000 and 60000;
```
- 참고: 엑셀 일련번호 46183 = 2026-07-06 (기준 `date '1899-12-30' + serial`).

## H. ★카드 매입 구매상품 건별 부분 선결제 (PR #131, #132, #134, #135, #136) — 오늘의 핵심
### 배경/요구
- 07/06 구매는 **상품 건별 카드구매(한도차감)** 로 매입 결재. 나중에 한도가 필요하면
  그 매입의 **구매상품을 건별로 골라 선결제**(앞당겨 결제)하여 **그 항목 금액만 한도 복구**.
- 캘린더 목적 = 자금계획(몇월 며칠에 얼마 빠지나). 선결제분은 선결제한 날 출금, 결제예정일엔 줄어든 나머지 청구.

### 최종 동작(합의)
- **선결제 처리 모달**(한도 현황 탭 → 💚 선결제 처리): 미결제 매입 목록 → 매입 **금액 클릭 시 구매상품 상세 펼침**
  → 앞당겨 결제할 상품만 체크 → 처리 시 확인창 → 그 항목에 `prepaid_date/prepaid_at` 기록.
  - 이미 선결제/취소된 항목은 회색으로 잠금(이중처리 방지). 상세 품목 없는 매입은 전체를 결제예정일 앞당김(`appr:` 폴백).
- **한도 현황**: `미결제 매입 = 총액 − 취소분 − 선결제분(선결제일이 오늘까지 지난 것만 복구)`. 자동 연동.
- **캘린더(자금계획)**:
  - 결제예정일 청구액 = `총액 − 선결제총액`(선결제분 이미 빠진 실제 청구액; 0이면 미표시)
  - 선결제일 = `+선결제금액`(그날 실제 통장서 나간 출금)
  - → 두 날 합치면 전체금액, 이중반영 없음.
  - 취소·환불은 기존대로 환불일에 `−`(실제 카드는 청구 후 환불 흐름).
- **선결제 이력**: `선결제처리` 로그(누가/언제/무엇/얼마/선결제일) — 모달 내 이력 + 변경 로그 탭.
- **주의(설계)**: 선결제일 **기준**. 오늘=즉시 복구, 미래날짜=그날 복구.

### 커밋 흐름
- #131 선결제 처리(매입 단위, 한도 복구) → #132 확인창+건별 정보+이력
- #134 **구매상품 건별 부분 선결제**(prepaid_date/prepaid_at) → #135 예정일 청구액에서 차감 → #136 선결제일 출금(+) 표기
- 오늘 품질점검에서 추가: **선결제된 항목은 취소 모달에서도 잠금**(총액−취소−선결제 이중차감 방지)
- 파일: `src/app/cards/CardsContent.tsx`, `db/approval_item_prepaid.sql`

### 실행 SQL (7/6 — 실행 완료)
```
alter table public.approval_items add column if not exists prepaid_date date, add column if not exists prepaid_at timestamptz; create index if not exists idx_approval_items_prepaid on public.approval_items (approval_id) where prepaid_date is not null;
```

## I. 품질 점검(선결제 기능 독립 리뷰 + 수정) — 7/6

    → ① 취소 모달에서 **선결제된 항목은 취소 잠금**(체크 불가, '선결제' 표시). ② 계산에서도 취소된 항목 id는 선결제 합산에서 제외(방어).
  - **[MED] 선결제일 > 결제예정일 방지**: 앞당겨 결제인데 선결제일을 결제예정일보다 뒤로 잡으면 자금흐름 왜곡 → 처리 시 경고·차단.
  - **[MED] total_amount NaN 방어**: 한도 계산에서 `(p.total_amount || 0)` 처리(누락 시 대시보드 전체 NaN 되던 것).
- 남긴 소소한 한계(인지·문서화, 지금은 미수정):
  - 상세품목 없는 매입의 `appr:` 폴백은 결제예정일만 앞당김(선결제일 즉시복구 아님, 새 예정일 지나면 복구) + 완료표시 없음.
  - 선결제 이벤트 캘린더 상세 모달 하단 합계는 매입 전체 총액 표시(클릭한 항목 금액과 다를 수 있음 — 표시상).
