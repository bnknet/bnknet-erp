## 1. 슬랙 개념 — 쉽게
- **워크스페이스** = 회사 전용 슬랙 공간(=회사 건물). 우리는 `Bang`. 직원을 초대하면 이 안에 다 모임.
  - 카톡과 차이: 카톡은 개인용·단톡방이 흩어짐. 슬랙은 회사 공간 하나 안에 주제별 방이 정리됨. 입·퇴사=계정 넣고 빼기.
- **채널** = 주제별 대화방(`#공지` `#매출` `#물류` `#cs`). 공개/비공개 설정 가능.
- **DM** = 1:1 개인 대화. (지금 대표님이 봇과 쓰는 방식)
- **앱/봇** = 공간에 들어와 일하는 도구. 우리 ERP 비서봇이 여기 해당.
- **역할(권한)**: 소유자(Owner=대표)/관리자(Admin)/멤버(직원). 소유자가 초대·설정 관리.

## 3. ★핵심 결정 — 봇을 누구에게 열까 (7/7 함께 결정)

### 옵션 A: 대표·실장만 (지금 상태 유지)
- 장점: **안전**. 매출·급여·결재 등 민감 데이터가 직원에게 노출될 걱정 없음. 천천히 검증.
- 단점: 직원 생산성 향상 효과는 아직 없음. 직원 문의는 여전히 대표님께 몰림.

### 옵션 B: 직원도 열되 "역할별 범위 제한"
- 봇 **하나**에서 역할별로 볼 수 있는 데이터를 다르게(권한 분기).
  - 예) 재고담당=재고/발주만, CS=주문상태/취소만. **매출·급여·원가·결재는 대표·실장만.**
- 장점: 직원이 카톡·전화 대신 봇에 직접 물어봐 **대표님 시간 절약**. "○○상품 재고 얼마?", "이 주문 취소됐어?" 등 셀프서비스.
- 단점: **권한 설계·테스트 개발이 더 필요**(누가 뭘 볼 수 있나 백엔드에서 강제). 지금은 화이트리스트=대표님만이라 이 분기 로직을 만들어야 함.

### 현재 추천 (논의 출발점 — 확정 아님)
1. **1단계(지금~2~3주)**: 대표·실장만으로 실사용하며 정확도·비용 검증.
2. **2단계**: 검증되면 직원에게 **범위 제한**으로 확대(옵션 B). 이때 역할별 도구 게이팅 개발.
- 이유: 처음부터 전원 열면 민감데이터 사고·비용·오답 리스크. 대표님이 먼저 신뢰도 확인 후 확장이 안전.
- ※ 이건 **출발점 제안**이고, 7/7에 대표님 상황(직원들이 지금 뭘 자주 묻는지 등) 들어보고 같이 결정.
