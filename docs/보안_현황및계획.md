# 보안 현황 및 계획 (2026-06-30 점검)

> 결정: **전면 보안 전환은 7/1 정식 가동 이후 별도 프로젝트로.** (가동 직전 전환은 라이브 중단 위험)
> 이 문서 = 현황·위험·계획 기록. 가동 안정화 후 이 문서 기준으로 진행.

## 1. 현재 보안 구조

| 항목 | 상태 | 평가 |
|---|---|---|
| API 키 | env 변수(`NEXT_PUBLIC_*`), repo 미커밋, `.env*` gitignore | ✅ 양호 (하드코딩 없음) |
| 로그인 | `employees` 테이블 email + `password_hash` **클라이언트 직접 비교** | ⚠️ 비번 평문 쿼리 전송 |
| RLS 정책 | 모든 테이블 `temp_anon_access` = `to anon using(true)` | 🔴 anon에게 전체 열림 |
| 권한 체크 | 화면(클라이언트)에서만 (role 분기, isPathAllowed) | 🔴 API 직접 호출 시 우회 |

## 2. 핵심 위험

- **anon 키는 앱 번들에 공개** → 그 키만 있으면 **로그인 없이 DB 전체 접근 가능**
  (직원 비밀번호, `accounts` 서비스 계정 비번, 카드정보, 매출 등 전부 읽기/쓰기)
- 로그인은 사실상 **"화면 잠금"** 이지 데이터 보호가 아님.
- 비밀번호가 평문으로 URL 쿼리에 실려 비교됨 (`password_hash=eq.{plaintext}`).

> 단, 현재는 **내부용 ERP + URL 비공개** 라 외부 노출 가능성은 낮음(베타 수준). 그래도 정식 운영 전 강화 필요.

## 3. 제대로 된 해결책 (7/1 이후 본 작업)

**Supabase Auth 전환 + RLS 재작성** — 큰 작업이라 단계적으로:

1. **Auth 도입**: 커스텀 로그인 → Supabase Auth(이메일/비번 or 매직링크). `employees`를 `auth.users`와 연결(매핑 테이블).
2. **JWT 기반 세션**: 로그인 시 발급된 토큰으로 API 호출 (anon 키 단독 접근 차단).
3. **RLS 전면 재작성**: 테이블별로 `auth.uid()`/role 기반 정책. `temp_anon_access` 전부 제거.
4. **권한 서버측 강제**: 민감 테이블(employees/accounts/cards)은 역할별 정책 + 필요한 건 security definer RPC로.
5. **비밀번호 해싱**: pgcrypto `crypt()` + 서버측 `verify_login` RPC.
6. **점진 전환·테스트**: 한 모듈씩 RLS 적용하며 동작 확인 (한 번에 다 바꾸면 위험).

## 4. (선택) 가동 직전에 할 수 있는 저위험 강화

> 시간/리스크 여유 있을 때만. 안 해도 7/1 가동엔 지장 없음.

- **로그인 RPC + 비번 해싱**: 로그인만 서버측 `verify_login(email,password)`(pgcrypto)로 교체 → 비번 평문 노출·직원테이블 통째 유출 위험 감소. (로그인 흐름만 건드려 비교적 안전, 단 테스트 필요)
- **Storage 버킷 공개범위 점검**: 첨부파일 버킷이 불필요하게 public인지 확인.

## 5. 트리거 (미래의 나에게)

- 사용자가 **"보안 작업하자"** 고 하면 → 이 문서 3번 단계대로 **7/1 가동 안정화 확인 후** 진행.
- ⚠️ RLS는 **한 모듈씩** 바꾸고 매번 라이브 동작 확인. `temp_anon_access`를 한꺼번에 지우지 말 것(전체 마비 위험).
- Auth 전환은 로그인·세션·모든 supabaseFetch 호출에 영향 → 충분한 테스트 환경에서 먼저.
