## 0. 항상 지켜야 할 3대 운영 규칙 (가장 중요 — 절대 누락 금지)

작업할 때마다 아래 세 가지를 매번 챙긴다.

### (1) 보안 — 코드에 안전장치를 기본으로 포함
- API 키 등 비밀값은 코드에 직접 쓰지 않고 환경변수(.env)로 분리한다.
- .env 파일은 .gitignore로 막아 GitHub에 절대 올라가지 않게 한다.
- 권한(RLS)은 모듈마다 빠뜨리지 않고 적용한다.
- 입력값 검증 등 기본 방어를 코드에 포함한다.
- 모듈 완성 직후, 실제 계정으로 권한이 제대로 막히는지 직접 테스트한다.
  (예: 일반직원 계정으로 로그인 → 재고 수정이 정말 막히는지 확인)

### (2) 백업 — 단계마다 점검
- 코드 백업: GitHub에 잘 저장됐는지 확인.
- DB 자동 백업: Supabase 자동 백업이 켜져 있는지 확인.
- 수동 백업: 한 달에 한 번 데이터를 파일로 내려받아 PC·외장하드에 보관.
- 데이터가 쌓이는 모듈(주문·재고 등)을 만들 때마다 백업을 짚는다.

### (3) 유료 전환 시점 안내
- 실시간 사용량은 직접 확인 불가하므로 정확한 "날짜"는 못 짚는다.
- 대신 유료 전환을 고려할 신호가 보이면 미리 알린다:
  - 직원들이 본격적으로 동시에 사용하기 시작할 때
  - 데이터가 많이 쌓이는 모듈을 붙일 때
  - 무료 백업 보관 기간이 회사 안전 기준에 부족해질 때
- 그 시점에 Supabase·Vercel 사용량 확인 방법을 안내하고, 결제는 운영자가 판단.

### ERP 통합 범위 (2026-06-17 결정)
- **기본: 사업자별 분리** → 매출/재고 모두 사업자 구분해서 집계 (세무신고·정산 편의)
- **전체 합산 보기**는 나중에 필요할 때 추가 기능으로 붙임
- DB 설계: 모든 테이블에 `company` 컬럼 추가 → 필터 하나로 사업자별/전체 전환
- 기본적으로 사업자별 취급 상품이 달라서 구분이 쉬움
- 예외: 쿠폰 계정 이슈 등으로 같은 상품을 두 사업자에서 각각 매입한 경우 있음
  → 재고는 **사업자 + 상품명** 조합으로 관리 (같은 상품이라도 사업자 다르면 별개 재고로 처리)
  → 초기 데이터 세팅 시 사업자별 재고 현황 방성훈 실장이 직접 공유 예정

### 현재 업무 툴 현황
| 용도 | 툴 |
|------|-----|
| 회사 정보·외부 자료 | Notion |
| 내부 파일 저장 (재고·매출·회계 등) | 윈도우 공유폴더 (사내 네트워크) |
| 직원 간 소통 | JANDI |
| 거래처 소통 | 네이버/지메일 회사 대표메일 |

→ ERP는 Notion·JANDI를 대체하지 않음. 윈도우 공유폴더의 엑셀 파일(매출·재고·정산)을 DB로 올리는 것에 집중.

### 현재 업무 방식 및 문제점

**매출 정리 (현재):**
- 사방넷 주문 로우데이터 다운 → 담당자가 수동으로 원가 입력 → ESM 들어가서 수수료 일일이 확인 → 수식으로 영업이익 산출
- 추이 볼 때마다 데이터 긁어서 새로 만들어야 함
- 담당자 개인 파일이라 부재 시 아무도 못 함
- 사업자별로 파일 따로 관리 중

**재고 관리 (현재):**
- 전날 재고 파일 열고 → 품목별 출고 수량 수동 입력 → 차감 → 퇴근 전 실사 크로스체크
- 재고 불일치 시 택배 픽업 중단, 맞춘 후 출고 (정확하나 시간 소요 큼)
- 반 아날로그식, 파일 수식 망가질 위험 있음
- 엑셀 파일 시트 구성: 재고 현황(자동계산) / 입고 관리 / 출고 관리

**재고 파일 구조 (2026-06-18 확인):**
- 품목 구성: BRAND + 상품명 + 코드(임의 순번)
- 재고 현황 시트: 코드 / 입고가 / 전일재고 / 최종재고 / 실제재고 / 총재고원가
- 입고 내역 시트: BRAND / 상품명 / 코드 / 날짜별 입고수량 / TOTAL
- 출고 내역 시트: BRAND / 상품명 / 코드 / 날짜별 출고수량 / TOTAL / 잔여재고

## 2. 현재 코드 상태 (2026-06-17 확인)

기존 bnknet-orders는 index.html 단일 파일로 된 가벼운 웹앱이다.

- 구성: 순수 HTML / CSS / JavaScript (빌드 도구 없음)
- 라이브러리: Google Fonts(Noto Sans KR), SheetJS(xlsx.js, 엑셀 처리)
- 데이터베이스: Supabase (REST API를 fetch로 직접 호출, JS SDK 미사용)
- 기능: 사방넷에서 받은 주문 엑셀을 업로드 → 자동 가공 → 새 엑셀로 다운로드.
  "DB에 저장" 시 중복 주문번호 체크 후 신규만 INSERT, "업로드 이력" 탭에서 조회.
- orders 테이블 작업 3가지:
  - GET /rest/v1/orders : 중복 주문번호 확인
  - POST /rest/v1/orders : 새 주문 일괄 INSERT
  - GET /rest/v1/orders : 업로드 이력 조회 (최신 200건)

### 보안 이슈 처리 현황 (2026-06-17 업데이트)
- ✅ orders 테이블 RLS 활성화 확인 + allow_all policy 삭제 → 외부 DB 접근 차단 완료
- ✅ GitHub 저장소 Private 확인 → 소스코드 외부 노출 차단 완료
- ✅ temp_anon_access policy 추가 (ALL / anon / true) → 테스트 기간 임시 DB 저장 허용 (2026-06-18)
- 🔲 index.html 137~138행 하드코딩된 키 제거 → ERP 신규 구축 시 처음부터 .env로 분리 예정
- 🔲 APP_PASSWORD 평문 제거 → ERP 로그인 시스템(Supabase Auth)으로 교체 예정

### 매출/정산 모듈 설계 방향 (2026-06-17 결정)

**문제:**
- 사방넷 주문 파일에는 판매가만 있고, 실제 수익을 알 수 없음
- G마켓/옥션(ESM)은 쿠폰 할인이 주문마다 달라서 수수료 분담금이 매번 다름
- 현재는 담당자가 ESM 시스템에 직접 들어가서 일일이 수동 확인 중 → 비효율

**해결 방향:**
- 사방넷 주문 파일 + ESM 정산 파일, 두 파일을 ERP에 업로드
- 주문번호 기준으로 자동 매칭 → 실제 정산금액 확인
- 원가/배송비/부가세 등 입력 후 영업이익 자동 계산

**저장할 항목 (확정 전 초안):**
- 판매가 (사방넷에서 가져옴)
- 실제 정산금액 (ESM 정산 파일에서 가져옴)
- 제품 원가
- 몰별 수수료
- 배송비
- 부가세
- → 영업이익 = 정산금액 - 원가 - 수수료 - 배송비 - 부가세

**업로드 주기 (안):**
- 사방넷 주문 파일: 매일 or 필요할 때
- ESM 정산 파일: 주 1회 or 월 1회 (정산 확정 후)

**사용 주체:**
- 매출관리 담당자가 ERP에 로그인해서 직접 파일 업로드 및 현황 조회
- 사장/관리자는 대시보드에서 전체 매출·이익 그래프 확인

**제공할 화면 (대시보드 항목):**

기간 현황
- 일 매출
- 주간 매출 / 전주 대비 비교
- 월 누적 매출 / 전월 대비 비교
- 연간 매출 / 전년 대비 비교

- 매출 현황 페이지 경로는 `/sales`(사이드바 "매출 현황" 메뉴)이며 현재 **빈 페이지(미구현)** 상태 → 신규 구축 대상.

**데이터 흐름:**
- 매출 출처: 주문 변환 → `orders` 테이블 (판매가=금액, 수량, 몰명, 날짜 보유).
- 매출 인식 시점: **주문 변환한 날 = 그날 매출** (당일/주간/월간/누적, 지난 기간 대비 비교, 추이 그래프).
- 원가 출처: **재고(inventory)의 개당원가(cost_price)** 를 주문 상품명(+사업자)으로 매칭 → 영업이익 자동 계산.
- 영업이익 = 판매금액 − 원가(− 수수료·배송비·부가세는 2차).

**원가 미입력 경고 (필수 — 데이터 정확도 핵심):**
- 판매됐는데 재고에 원가가 없거나 상품명이 안 맞아 원가를 못 끌어온 품목은 **영업이익이 비게 됨**.
- → 매출 현황 화면 상단에 **경고 배너 필수 노출** (예: "⚠️ 원가 미입력 상품 N건 — 영업이익 미반영").
- [확인] 클릭 시 목록 창(상품명/사업자/판매건수·금액) → **그 자리에서 원가 바로 입력·저장(재고 반영)** → 매출 즉시 재계산. (다른 화면으로 안 넘어가고 즉시 수정)
- 목적: 데이터 꼬임 방지 + 정확한 이익 계산 + 자연스러운 재고 원가 정비.

**안전장치 (자동화엔 반드시 실패 감지·알림 — 재고와 동일 철학, 2026-06-24 결정):**
- 원칙: **자동으로 돌아가는 기능은 "잘못되면 바로 알 수 있는 장치"를 항상 함께 둔다.** (재고 자동저장에 적용한 방식 그대로 매출에도 적용)
- 매출은 `orders`(주문 변환 저장분)에서 집계 — orders는 날짜가 박힌 append-only 기록이라 "지난달 대비/추이"는 실시간 집계로 가능(재고처럼 일일 스냅샷이 꼭 필요하진 않음). 단, 만약 일일 매출 스냅샷을 두기로 하면 재고와 동일하게 자동저장+누락감지+수동저장+이메일 경고를 함께 붙인다.
- **화면 경고 배너**: 이상 징후(예: 오늘 주문은 저장됐는데 매출 집계가 0, 또는 원가 미입력 다수) 시 매출현황 상단 + 대시보드에 경고.
- **이메일 알림 재사용**: 재고에 구축한 인프라(Resend + Supabase `pg_net` + pg_cron, 발송 bpride5073@gmail.com)를 **그대로 재사용** — 매출 관련 자동 점검 실패 시에도 메일. 새 점검 함수 + cron만 추가하면 됨(메일 발송 경로는 이미 완성).
- 매출 탭 개발 시 이 안전장치를 **함께 구현**한다(나중에 따로 붙이지 말 것).

**★ 최우선 원칙 — 재고·매출·주문건 3중 정합성 (항상 일치, 절대 어긋나지 않게):**
- **매출은 별도 합계로 저장하지 않고 `orders`(상태 기준)에서 항상 실시간 집계** → 주문 취소 시 매출 자동 제외, 주문건↔매출이 구조적으로 어긋날 수 없음.
- 취소/반품 처리는 **하나의 DB 트랜잭션(Supabase 함수/RPC)** 으로: 주문 상태 변경 + 재고 수량 복구 + 입출고 로그를 한 묶음 → "반쪽만 반영"되는 불일치 원천 차단(모두 성공 또는 모두 롤백).
- **중복 취소/복구 방지**: 이미 취소된 주문 재취소 불가(재고 이중 복구 방지). 부분 취소는 누적 취소수량이 주문수량을 못 넘게.
- 변환 시 차감량을 주문 단위로 기록 → 복구는 정확히 그 값만큼.
- **정합성 점검 안전장치**: 주문 상태와 재고 복구가 어긋나면 경고 배너 + 이메일(기존 인프라 재사용).

**주문 변환 → 재고 자동 차감 + 취소·반품 되돌리기 (2026-06-24 확정):**
- 7/1 가동 흐름: 오전 주문 변환·DB 저장 시 → **매출 집계 + 각 주문 상품 수량만큼 재고 자동 차감**(출고 로그 기록). 주문 상품명을 재고 품목(상품명+사업자)에 매칭. ※ 기존 "재고는 출고/실사 때 수기 차감" → "변환 시 자동 차감"으로 전환되는 신방식.
- 매칭 안 되는 상품(재고에 없음/이름 불일치)은 자동 차감 불가 → **미매칭 경고 배너**(원가 미입력 경고와 동일 메커니즘)로 표시.
- **차감량을 시스템이 기록** → 취소 시 정확히 그만큼만 되돌릴 수 있게.
- **취소 처리(출고 전)**: ERP에서 주문번호/수취인으로 검색 → "취소"(사유·날짜). 자동으로 ① 매출에서 제외(취소율 반영) ② 재고 +복구. **부분 취소 지원**(취소 수량 입력 → 전체 취소도 같은 화면에서 처리).
- **재고 취소 이력 필수 기록**: 재고 +복구는 입출고 내역(`inventory_logs`)에 구분="취소", **처리 날짜(created_at)·처리 담당자(created_by=로그인 사용자)·사유**를 함께 남겨 "○월○일 / 담당자 / 취소 / +N개 / 사유" 형태로 추적 가능하게. (기존 로그 구조로 지원됨)
- **반품 처리(출고 후)**: "반품"으로 처리 → 매출 차감 + 재고 +복구(취소와 동일 메커니즘, 사유 구분).
- ⚠️ 이 기능은 **매출 모듈과 한 세트로 함께 구현**(변환 시 자동 차감이 없으면 취소 되돌리기도 의미 없음). 취소/반품은 삭제가 아니라 **상태(정상/취소/반품) 표시**로 처리해 기록 보존.
